# Platform Detection: Mac Mini vs VPS Docker

BlackCEO Command Center runs on two deployment targets:

  1. The operator's Mac Mini (native macOS install, primary development host)
  2. A Hostinger VPS Docker container (per-client production install)

The two hosts differ in where the OpenClaw config lives, where the vault lives, where per-agent scratch directories live, and which bootstrap script ran at install time. Rather than hardcoding paths everywhere, the codebase routes all path resolution through a single helper module at [`../src/lib/platform.ts`](../src/lib/platform.ts).

This document explains how detection works, what each helper returns, how to override behavior, and what the differences look like in practice.

## The detection function

```ts
export type Platform = 'mac-mini' | 'vps-docker';

export function detectPlatform(): Platform
```

Source: [`../src/lib/platform.ts`](../src/lib/platform.ts).

Detection order (highest precedence first):

  1. `OPENCLAW_PLATFORM` env var, when set to `mac-mini` or `vps-docker`. This is the operator escape hatch for tests, CI, or atypical hosts (for example, running the VPS container image locally on a Mac for debugging).
  2. Presence of the `/data/.openclaw` directory. Hostinger's Docker template mounts `/data` as the persistent volume, so this directory reliably exists on every VPS install and never exists on a fresh Mac.
  3. Default to `mac-mini`. The Mac is the historical baseline and the safer fallback because if you guess wrong and code tries `/data` paths on a Mac, you fail loudly; if you guess wrong the other way, the Mac home directory paths exist on the VPS too because of how the OpenClaw container is provisioned, so the failure is also obvious but more recoverable.

The result is NOT cached. Detection is cheap (one env read plus at most one `existsSync`) so it is safe to call on every request when convenient. Hot paths should still cache locally.

## Per-platform path helpers

The module exposes three helpers. Every code path that needs to touch a long-lived directory MUST go through one of them.

### openclawConfigPath

```ts
export function openclawConfigPath(): string
```

Absolute path to `openclaw.json`. This is the canonical OpenClaw configuration file the dashboard reads at boot to learn the gateway URL, gateway token, and skill registry.

  - Mac Mini: `~/.openclaw/openclaw.json` (resolved via `os.homedir()`)
  - VPS Docker: `/data/.openclaw/openclaw.json`

### vaultRoot

```ts
export function vaultRoot(): string
```

Root directory for the operator's vault. Goals, Journal entries, Memory-indexed markdown, Studio outputs, Research results, and Web Agent transcripts all write underneath this root.

  - Mac Mini: `~/clawd/` (the historical operator vault, also used by OpenClaw skills)
  - VPS Docker: `/data/.openclaw/workspace/`

The trailing separator is included in the return value so callers can `path.join(vaultRoot(), 'Goals', 'mygoal.md')` without thinking about it.

### operatorScratchRoot

```ts
export function operatorScratchRoot(): string
```

Root directory for per-agent scratch space. Bridge sessions, Workspace file storage by-agent, and any CLI agent's working directory all live under this root.

  - Mac Mini: `~/clawd/scratch/`
  - VPS Docker: `/data/.openclaw/scratch/`

Per-agent subdirectories live underneath this root, one directory per CLI agent slug (`claude`, `codex`, `antigravity`, `hermes`, `gemini`, `fcc`, `openclaw`). Callers typically do `path.join(operatorScratchRoot(), agentSlug)`.

## Persona file lookup

The dashboard surfaces personas (IDENTITY.md, SOUL.md, AGENTS.md, TOOLS.md, USER.md, MEMORY.md) generated by Skill 23 (AI Workforce Blueprint). The lookup walks these locations in order on either platform:

  1. `<vaultRoot()>/.openclaw/agents/<agent-slug>/` (preferred, written by Skill 23)
  2. `<vaultRoot()>/agents/<agent-slug>/` (legacy location, supported for v3.x compatibility)
  3. `<openclawConfigPath dir>/agents/<agent-slug>/` (operator override)

On Mac Mini this resolves to paths under `~/clawd/...`. On VPS Docker it resolves to paths under `/data/.openclaw/workspace/...`. The runtime persona resolver (PRD Section: Runtime Persona Resolution) uses the first match. If none exist, the dashboard renders a placeholder card and surfaces a "Run Skill 23" call to action.

## Env variable overrides

Two env vars override the defaults. Both are useful for test rigs, custom Hostinger image layouts, and unusual operator setups.

### OPENCLAW_DATA_DIR

Overrides the OpenClaw data root. When set, `openclawConfigPath()` returns `<value>/openclaw.json` regardless of platform.

Typical use: a developer wants to run two VPS-style instances on the same Mac, each pointing at its own copy of OpenClaw config. Set `OPENCLAW_DATA_DIR=/tmp/openclaw-instance-a` for one and `/tmp/openclaw-instance-b` for the other.

### WORKSPACE_BASE_DIR

Overrides the vault root. When set, `vaultRoot()` returns `<value>` regardless of platform. The scratch root is NOT overridden by this; if you also want to relocate scratch, set both.

Typical use: an operator with a large external drive on the Mac wants the vault on the drive instead of the home folder. Set `WORKSPACE_BASE_DIR=/Volumes/ExternalSSD/clawd`.

Both overrides are read fresh on every call to the helper. There is no boot-time cache. This is intentional so that the operator can change the env vars at runtime (via the `/api/system/bootstrap` re-run flow) and have the change take effect for the next request without a process restart.

## Disk probe behavior

The system status panel reports disk free space for the platform's primary volume. The probe behavior differs because the relevant volume differs.

### Mac Mini

  - Probe target: `os.homedir()` (resolves to the operator's home volume, normally the boot APFS volume).
  - Threshold: warn at less than 10 GB free, degrade at less than 1 GB free.
  - Implementation: `df -k $HOME | tail -1`, parse the available column.

### VPS Docker

  - Probe target: `/data` (the Hostinger persistent volume mount).
  - Threshold: warn at less than 2 GB free, degrade at less than 500 MB free. Lower thresholds because VPS volumes are smaller and the Section 6.2 pre-flight check already requires at least 500 MB free.
  - Implementation: `df -k /data | tail -1`, parse the available column.

The probe is wired into the System Status aggregator alongside the CLI probe, Cloudflare Tunnel probe, and Cloudflare Access probe (see P1-12).

## Bootstrap script choice

The bootstrap script differs by platform. Both live at [`../scripts/install/`](../scripts/install/).

### Mac Mini: `mac-mini-bootstrap.sh`

Installs:

  - Homebrew (if missing) and brew packages: node, python, uv, ffmpeg, jq
  - PM2 globally via npm
  - Operator CLIs: Claude Code, Codex, Antigravity, Hermes, Gemini, OpenClaw
  - Obsidian (Mac Mini only, because the vault is intended for desktop editing)
  - Playwright browser dependencies for the Web Agent sub-module
  - cloudflared and registers the Cloudflare Tunnel for the operator's subdomain
  - Initializes `~/clawd/` and `~/.openclaw/` if missing
  - Writes the env file at `~/Documents/<client-slug>/.env` or whichever canonical Mac location the bootstrap uses

The script can be re-run safely; it is idempotent on every step.

### VPS Docker: `vps-docker-bootstrap.sh`

Designed to run INSIDE the OpenClaw Docker container, not on the bare VPS host. Installs:

  - Linuxbrew packages: node, python, uv, ffmpeg, jq (because Hostinger's `apt` shim redirects to brew anyway, see memory entry `feedback-hostinger-apt-is-a-brew-shim`)
  - PM2 globally via npm into `/data/.npm-global` (so that PM2 persists across container recreate, see memory entry `openclaw-vps-docker-persistent-upgrade`)
  - Operator CLIs into the container
  - Playwright browsers
  - Initializes `/data/.openclaw/workspace/` and verifies `/data/.openclaw/` exists
  - Writes the env file at `/data/.openclaw/.env` (canonical) and falls back to other documented locations if the operator overrode

The script MUST run inside the container because `/data` only exists inside. Running it on the bare host would do nothing useful.

The bootstrap chooser at [`../src/app/api/system/bootstrap/route.ts`](../src/app/api/system/bootstrap/route.ts) (P1-13) calls `detectPlatform()` first and spawns the right script. Re-run is exposed in the System Status drawer.

## Sanity checks during development

When debugging a path-related issue, run this in a Node REPL or a small script:

```ts
import { detectPlatform, openclawConfigPath, vaultRoot, operatorScratchRoot } from './src/lib/platform';

console.log('platform:', detectPlatform());
console.log('config:  ', openclawConfigPath());
console.log('vault:   ', vaultRoot());
console.log('scratch: ', operatorScratchRoot());
```

If the platform value is wrong, check:

  1. `echo $OPENCLAW_PLATFORM` (should be empty unless you intentionally set it)
  2. `ls -la /data/.openclaw 2>&1` (exists -> VPS detection wins)
  3. `echo $OPENCLAW_DATA_DIR` and `echo $WORKSPACE_BASE_DIR` for path overrides
